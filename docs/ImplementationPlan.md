# ImplementationPlan


## 1. 목표

PlanMate를 **문서 → MVP 구현 → Azure 배포** 순서로 진행한다.  
우선순위는 “완성 가능한 범위의 MVP”를 빠르게 만드는 것이다.

## 2. 개발 원칙

- 한 번에 모든 기능을 넣지 않는다.
- 캘린더 조회와 반복 일정은 핵심 기능이므로 초반에 구조를 제대로 잡는다.
- UI 예쁘게 다듬기보다 먼저 데이터 흐름을 완성한다.
- Azure 배포는 마지막이 아니라 중후반에 미리 시도한다.

## 3. 단계별 계획

## Phase 0. 프로젝트 초기화
### 목표
- 개발 환경 및 기본 구조 확정

### 작업
- Next.js 프로젝트 생성
- TypeScript strict 설정
- Tailwind CSS 설정
- ESLint / Prettier / lint script 정리
- 기본 폴더 구조 생성
- `.env.example` 작성

### 완료 기준
- 로컬에서 앱 실행 가능
- 기본 빈 화면 렌더링 가능
- GitHub 저장소 초기화 완료

---

## Phase 1. DB 설계 및 연결
### 목표
- Azure SQL Database와 앱 연결

### 작업
- Azure SQL Server / Database 생성
- 방화벽 및 접속 설정
- DB 연결 유틸 작성
- `events`, `event_occurrences` 테이블 생성
- 연결 테스트 API 작성

### 완료 기준
- 로컬에서 DB 연결 성공
- 간단한 select/insert 테스트 통과

---

## Phase 2. 이벤트 CRUD API
### 목표
- 일정 생성/수정/삭제 API 완성

### 작업
- `POST /api/events`
- `PATCH /api/events/:id`
- `DELETE /api/events/:id`
- 입력 검증(zod 등) 적용
- 표준 응답 포맷 정의
- 에러 처리 공통화

### 완료 기준
- Postman/브라우저 기준 CRUD 정상 동작
- 잘못된 입력값에 대해 에러 응답 제공

---

## Phase 3. 범위 조회 + 반복 일정 로직
### 목표
- 일/주/월 뷰용 범위 조회 API 구현

### 작업
- 범위 계산 유틸 작성
- `GET /api/events/range`
- 반복 일정 전개 로직 구현
- 완료 상태 조합 로직 구현
- 범위 조회 단위 테스트 작성

### 완료 기준
- 단일 일정과 반복 일정 모두 범위 조회 가능
- 월/주/일 기준 날짜 범위 계산 정확성 확인

---

## Phase 4. 메인 UI 구현
### 목표

## Phase 5. 일정 입력 UX 구현

### 완료 기준

### 완료 기준

## 6. 테스트 체크리스트

- 단일 일정 생성 가능
- 반복 일정 생성 가능
- 월별 뷰에 일정 표시
- 주별 뷰에 일정 표시
- 일별 뷰 상세 목록 표시
- 특정 발생일 완료 처리 가능
- 삭제 후 다시 조회해도 제거됨
- 모바일에서 일정 추가 가능
- 배포 환경에서 DB 연결 정상

## 7. 리스크 관리 포인트

- 반복 일정은 반드시 단위 테스트 작성
- Azure 배포는 기능 완성 직전이 아니라 중간에 한 번 먼저 시도
- 날짜 계산 유틸은 컴포넌트 안에 흩어놓지 말 것
- DB 쿼리와 UI 수정은 동시에 크게 바꾸지 말 것
