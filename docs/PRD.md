# PRD


## 1. 제품 개요

- **가칭:** PlanMate
- **한 줄 설명:** 일별 할 일을 추가하고, 이를 **월별 / 주별 / 일별 캘린더 뷰**로 확인할 수 있는 개인 일정 관리 웹앱
- **핵심 가치:** 단순 목록형 To-do가 아니라, **날짜 맥락 안에서 할 일을 계획하고 반복 습관까지 관리**할 수 있도록 돕는다.
- **배포 목표:** Azure App Service에 웹앱 배포, Azure SQL Database에 데이터 저장

## 2. 문제 정의

기존의 단순 To-do 앱은 “무엇을 해야 하는지”는 보여주지만, “언제 해야 하는지”를 직관적으로 보여주지 못하는 경우가 많다.  
특히 대학생이나 개인 사용자 입장에서는 다음과 같은 불편이 있다.

- 하루 단위로 할 일을 추가해도 전체 주간/월간 흐름이 잘 보이지 않는다.
- 반복 일정(운동, 공부, 복습, 회의 준비)을 매번 직접 다시 입력해야 한다.
- 목록형 UI는 일정 밀집도나 특정 주간의 부담을 파악하기 어렵다.

## 3. 목표

### 3.1 비즈니스/포트폴리오 목표
- Azure 기반 웹앱 + 데이터베이스 아키텍처를 직접 구현한다.
- CRUD, 캘린더 조회, 반복 일정 처리라는 핵심 기능을 포함한 완성형 프로젝트를 만든다.
- 학생 포트폴리오로 설명 가능한 수준의 구조화된 앱을 완성한다.

### 3.2 사용자 목표
- 특정 날짜에 할 일을 쉽게 등록할 수 있어야 한다.
- 일별 / 주별 / 월별로 일정을 자연스럽게 탐색할 수 있어야 한다.
- 반복 일정 등록으로 반복 입력 부담을 줄일 수 있어야 한다.
- 완료 여부를 직관적으로 체크할 수 있어야 한다.

## 4. 타깃 사용자

### 1차 타깃
- 개인 일정/할 일을 관리하고 싶은 대학생
- 주간 계획과 월간 계획을 함께 보고 싶은 사용자
- 공부 루틴, 운동 루틴, 과제 일정을 같이 관리하고 싶은 사용자

### 2차 타깃
- 포트폴리오용 데모 앱을 보는 면접관/멘토
- Azure 기반 CRUD 프로젝트 예시를 확인하려는 개발 입문자

## 5. 제품 범위

## 5.1 MVP 범위
- 일정(할 일) 생성
- 일정 수정
- 일정 삭제
- 일정 완료 처리
- 일별 / 주별 / 월별 보기
- 반복 일정 생성
  - 매일
  - 매주
  - 매월
- 선택한 날짜 기준으로 일정 목록 확인
- Azure SQL Database 저장
- Azure App Service 배포

## 5.2 MVP 제외 범위
- 사용자 로그인/회원가입
- 협업 / 공유 캘린더
- 알림 푸시 / 이메일 리마인더
- 드래그 앤 드롭 일정 이동
- 개별 반복 항목만 수정/예외 처리
- 파일 첨부
- 태그/카테고리 다중 분류

## 6. 핵심 사용자 시나리오

### 시나리오 A: 단일 일정 등록
사용자는 특정 날짜를 클릭하고 “알고리즘 과제 제출” 같은 할 일을 추가한다.  
추가 직후 해당 날짜 셀/주간 뷰/일간 목록에 일정이 표시된다.

### 시나리오 B: 반복 일정 등록
사용자는 “운동” 일정을 매주 월/수/금 반복으로 등록한다.  
앱은 선택한 날짜 범위 내에서 반복 규칙에 맞는 항목을 캘린더에 보여준다.

### 시나리오 C: 월별 계획 확인
사용자는 이번 달 전체를 보면서 어느 날짜에 일정이 몰려 있는지 확인한다.  
특정 날짜를 선택하면 그날의 상세 일정 목록을 별도 패널에서 확인할 수 있다.

### 시나리오 D: 완료 체크
사용자는 오늘 해야 할 항목을 끝낸 뒤 완료 체크를 한다.  
완료 상태는 UI에 즉시 반영되고, 이후 같은 범위를 다시 조회해도 유지된다.

## 7. 사용자 스토리

- 사용자는 날짜를 클릭해서 새로운 할 일을 추가할 수 있다.
- 사용자는 오늘, 이번 주, 이번 달의 일정을 각각 다른 관점에서 볼 수 있다.
- 사용자는 반복되는 루틴을 매번 입력하지 않고 한 번만 설정할 수 있다.
- 사용자는 완료한 일정과 미완료 일정을 구분해서 보고 싶다.
- 사용자는 모바일에서도 핵심 기능을 사용할 수 있기를 원한다.

## 8. 기능 요구사항

### FR-01 일정 생성
- 제목은 필수 입력값이다.
- 날짜는 필수 입력값이다.
- 설명은 선택 입력값이다.
- 반복 유형은 `없음 / 매일 / 매주 / 매월` 중 선택 가능하다.
- 반복 종료일은 선택 입력값이다.

### FR-02 일정 조회
- 사용자는 일별 / 주별 / 월별 범위로 일정을 조회할 수 있어야 한다.
- 선택한 기준 날짜(anchor date)에 따라 범위가 계산되어야 한다.
- 월별 뷰에서는 월 달력 형태가 보여야 한다.
- 주별 뷰에서는 7일 묶음으로 일정이 보여야 한다.
- 일별 뷰에서는 선택한 날짜의 상세 목록이 보여야 한다.

### FR-03 일정 완료 처리
- 단일 일정은 해당 날짜 기준으로 완료 처리할 수 있어야 한다.
- 반복 일정은 특정 발생일(occurrence) 기준으로 완료 상태를 저장해야 한다.

### FR-04 일정 수정/삭제
- 사용자는 기존 일정을 수정할 수 있어야 한다.
- 사용자는 기존 일정을 삭제할 수 있어야 한다.
- MVP에서는 반복 일정 수정 시 “시리즈 전체 수정”만 지원한다.

### FR-05 상태 피드백
- 생성/수정/삭제/완료 처리 시 성공 또는 실패 메시지를 제공해야 한다.
- 로딩 상태와 빈 상태를 UI에서 구분해서 보여줘야 한다.

## 9. 비기능 요구사항

- **성능:** 월별 조회 시 체감상 빠르게 렌더링되어야 한다.
- **신뢰성:** 새로고침 후에도 저장된 데이터가 유지되어야 한다.
- **확장성:** 추후 로그인/다중 사용자 기능을 추가할 수 있어야 한다.
- **보안:** 클라이언트에서 DB에 직접 접근하지 않고 서버 API를 통해서만 접근해야 한다.
- **유지보수성:** 일정 규칙, API, DB 구조가 문서화되어 있어야 한다.
- **반응형:** 모바일과 데스크톱 모두에서 사용할 수 있어야 한다.

## 10. 성공 기준

### MVP 완료 기준
- 일정 CRUD가 정상 동작한다.
- 일/주/월 캘린더 탐색이 가능하다.
- 반복 일정이 범위 조회에 반영된다.
- 완료 처리 결과가 다시 조회해도 유지된다.
- Azure App Service에서 배포 URL로 앱이 정상 동작한다.
- Azure SQL Database와 연결된 상태로 운영 가능하다.

### 품질 기준
- 치명적 런타임 에러 없이 주요 흐름이 동작한다.
- 모바일에서 일정 추가와 조회가 가능하다.
- 날짜/반복 로직이 문서와 일치한다.

## 11. 리스크와 가정

### 가정
- 초기 버전은 **개인용 / 단일 사용자 데모**를 목표로 한다.
- 시간 단위 스케줄링보다 **날짜 중심 일정 관리**를 우선한다.
- 반복 일정의 예외 편집은 MVP 범위 밖이다.

### 리스크
- 반복 일정 로직이 가장 구현 난도가 높다.
- 캘린더 라이브러리와 커스텀 데이터 구조 간 매핑이 꼬일 수 있다.
- Azure 배포 시 환경변수/DB 연결 설정에서 초기 시행착오가 발생할 수 있다.

## 12. 릴리즈 판단

아래 항목이 모두 충족되면 MVP 릴리즈 가능 상태로 본다.

- 캘린더 메인 화면 정상 렌더링
- 일정 생성/수정/삭제/완료 정상 동작
- 월별/주별/일별 전환 정상 동작
- 반복 일정 조회 정상 반영
- Azure 웹 배포 완료
- 운영 환경 DB 연결 완료
